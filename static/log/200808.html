タスクリストに要素を追加するとき

ゲームでタスクリストにタスクを作って格納するなんてことは普通にやると思います．

自分のソースでは，

Task* p = new Task;
taskList.push_front(p);

って書いてます．ちなみにタスクリストの全解放を行う機能も用意されています．何らかの例外がどこかで発生したときとゲーム自体の終了時は，それを呼び出す仕組みになっています．

for (std::list<Task*>::iterator it = taskList.begin(); it != taskList.end(); ++it) {
　delete *it;
}
taskList.clear();

って感じの．

けど今日のバイト中，もし push_front を行う場所で例外が発生したら，余裕でメモリリークしねぇ？これ？と思いました．

push_front メソッドで例外を意図的に発生させる方法なんぞ分からないので実験はしてませんが，おそらく new して push_front が中断されて catch ブロックに飛ぶのだから，このとき p が指す実体のデストラクタは呼ばれないはずですよね…？

解決策は，

1. スマートポインタを使う

2. 例外処理をその都度埋め込んでやる
Test* p = new Test;
try {
　taskList.push_front(p);
}
catch (exception& e) {
　delete p;
　throw e;
}
つまり↑のように逐一書く．

の 2 つがあると思います．

やっぱりスマポったら最強ね．

Twitter 導入について誰もが疑問に思うこと

mixi でエコーと呼ばれるサービスが始まりました．まぁ今更言うまでも無いですがどう見ても Twitter のパクリです．Twitter の面白さがようやく日の目を見始めてきた，ということでしょう．

で，何人かこれを使って楽しんでいる人たちが居ますが，その人たちには是非本家 Twitter も使ってもらいたいです．

しかし，この Twitter というサービスは，大抵誰しも最初は「何をすれば面白いのか分からない…」という疑問を抱きます (これは私も含む)．で，そのままずっと放置して，気が付けば最終発言が数ヶ月前…　という事態に陥ってしまうのです．実際私も初めて投稿してから一ヶ月近く放置していました．

この原因は，実はぶっちゃけ一つしかありません．フォローしてる人が少ないからなんです．

実際，私はあるとき勇気を出して数人をフォローしてみたところ，そのままあれよあれよと言う間にのめり込んで行ってしまいました．部活の存在を知ってフォローしてる人が 100 人を超えた辺りから，Twitter の本当の楽しさが見えてきたのです．

ということで，Twitter は何が楽しいのか分からなかった…　と仰る皆さん，是非，どんどん他人をフォローしてみて下さい．その人の発言を流し読みしてみて，ああ，趣味が合う部分があるな，と感じたら，もうそれだけで十分フォローしてみる価値はあります．

※そして Twitter は別にリムーブ (フォローしてる人を外すこと) をしてもいいよ，という文化があります．いざフォローしてみて，ああこの人の発言は見たくないな…　と思ったら，リムーブしても別に構わないのです．

一応，Twitter 初心者向けの参考記事へのリンクも貼っておきます．よければ合わせてご覧下さい．

コーディング規約

自分内でまたこれにちょっとした変化があります．

まず，ゲーム等においてタスク等を更新する際，今までは renew というメソッド名を用いていたのですがこれを他の方が使っているのを見たことがありませんｗ　ことごとく皆さん update を使ってらっしゃいます(^^;　なので，自分もそちらに切り替えることにしました．

また，コーディング規約の会というサイトからのリンクにある翔ソフトウェア (Sho's) - XP (エクストリーム プログラミング) - C++ コーディング標準に色々な単語の名前の対称性に関する記述がありますが，これにも厳格に従うことにします．

いままでタスクの起動と終了には start/end という名前を使っていたのですが，これからは start/stop を使うことにします．しかしうちのゲームのアルゴリズムでは，あくまでタスクが終了出来る条件を満たしたかどうかをこちらで判断して，そしてこちらからの指定で終了する，という処理にしてあるので，こちらから無条件に停止させるような意味を持つ (と少なくとも私は感じている) stop という単語を使うのには少々抵抗があります．というより，抵抗があったので今まで end を使っていました．しかしここも一人で突っ張っても仕方ないので，stop に慣れることにします．

あとそれに伴い，終了判定を行うメソッド名も変更します．今までは「終了しているか否か？」という意味合いを持たせた，boolean 値を返す isEnd というメソッドだったのです．また，これもよく考えると「タスクリストの末端要素か否かを判断する」という意味に取れなくも無いので，それもあって変更します．「停止させることが出来るか否か？」という意味合いを持たせた，boolean 値を返す canStop というメソッド名に変更します．

[2008-08-29T01:41 追記]

またよく考えてみたら，stop メソッドは別にいつ呼んでも構わないことに気づきました．canStop が false を返すから呼んではいけない，という存在ではありません (まぁ現状ではこれが true を返すときにしか stop させてないのですが…)．だから canStop というよりはやっぱり「終了しているか否か」を考えるのが本質なわけで．言い換えると「これ以上 update を実行しても意味がない」ということをあらわしているわけで．しかし前述の通り isEnd は意味が複数あると言えなくもないので，とりあえず hasFinished を使おうと思います．スペル長いけどね… orz

命名規約は読み易さとともに書き易さも補助してくれることが多いので，常にしっかりと意識しておきたいです．

メンバ関数テンプレート覚書

テンプレートクラスで，メンバ関数にまた別のテンプレートを使いたいとき，メンバ関数テンプレートという以下のような機能を使う．

template <typename T>
class Test {
public:
　//↓メンバ関数テンプレート
　template <typename U>
　void func(U x) {
　　//色々な処理
　}
　void func() {} //特に関係ないけど後の説明のために必要なメソッド
};

これは直感的に分かると思う．さて，ここでこのメンバ関数の定義部分が長くなってきて抜き出したいとする．

通常のメンバ関数であれば，

template <typename T>
class Test {
public:
　void func();
};

template <typename T>
void Test<T>::func()
{
}

とすればよいので，この考え方をそのままメンバ関数テンプレートに適用すると，以下のようになるはずである．

template <typename T>
class Test {
public:
　template <typename U>
　void func(U x);
};

template <typename T>
template <typename U>
void Test<T>::func(U x)
{
　//色々な処理
}

そして実際その通りである．typename T と typename U の行の順番はこれで納得いくはずである．納得いかない人はもっとよく考えるように．

ちなみに Borland C++ 5.5.1 で実験して確認した．
